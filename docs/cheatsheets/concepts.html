What Are CDEs?

Common Data Elements — standardized data fields from the NIH/NLM CDE Repository. Each CDE defines a reusable question, variable, or measurement used across clinical studies.

CDE Record Structure

  • tinyId — unique short identifier (e.g. "XyZ4k2")
  • designations[] — alternative names / display labels
  • definitions[] — explanatory text descriptions
  • valueDomain — permitted values, data type
  • sources[], properties[] — provenance metadata

Key Field Paths

designations.*.designationCDE names
definitions.*.definitionCDE descriptions
valueDomain.permissibleValues.*.valueMeaningNameValue labels
valueDomain.permissibleValues.*.valueMeaningDefinitionValue defs

Algorithms & Patterns

K-mer Phrase Mining

Iterative descending extraction of shared text phrases from CDE fields.

Algorithm

for k = k_max down to k_min:
  phrases = extract_kmers(texts, k)
  phrases = filter(freq ≥ n, tinyids ≥ t)
  emit(phrases)
  texts = mask(texts, phrases) # prevent re-counting

Key Parameters

-Kk_max (default 25)Longest phrase
-kk_min (default 3)Shortest phrase
-nfreq_minMin frequency
-tmin_tinyidsMin distinct CDEs
-Dde BruijnExtend via overlap graph
-SsubsumptionRemove subsumed

Masking

After each k, detected phrases replaced with placeholders. Prevents redundant shorter fragments at lower k values.

Dedup Pre-Pass

Before k-mer mining (Stage 0):

  1. Hash field texts — group identical strings shared by N+ CDEs
  2. Filter — only emit phrases with tokens > k_max
  3. No masking — sub-phrases remain visible to k-mer loop

Output: separate dedup_phrases.tsv. Short duplicates found naturally by k-mer mining.

Anchor Expansion

Context prefix/suffix expansion for instrument discovery:

  • Defined in config/anchor_expansions.yaml
  • Generates “as part of <X>”, “using the <X>”, etc.
  • Instrument-specific only — phrase/temporal patterns use \b word boundaries instead

Phase 1 uses anchor expansion; Phase 2 phrases are self-contained.

Subsumption

Removing redundant shorter patterns covered by longer ones.

Text-Based Subsumption

Pattern A subsumed by B when:

  • A is a substring of B, AND
  • every tinyId of A is covered by union of tinyIds of all longer patterns containing A

Empirical Validation

Per-tinyId, per-field verification after coalescing:

  • Check if short pattern matches text not already covered by any longer group member
  • Keep only if it matches independently
pattern_util -V coalesced.tsv \
  -i input.json -o validated.tsv

Coalescer Phases

Three-phase consolidation (flexible_pattern_matcher.py):

Phase 1a — Prefix-Kept

Keep shorter A alongside longer B when B.startswith(A) and A has tinyIds not covered by B.

Phase 1b — NP-Continuity

If extension tail is NP-like (80%+ Title Case words), keep B intact — preserves instrument sub-domain names.

# Kept: "Neuro-QOL Sleep Disturbance"
# Rolled down: "PHQ-9, which is a"

Phase 2 — Prefix Trie

Group patterns sharing common prefix. Merge groups where shorter prefix covers same tinyIds.

Temporal Variant Expansion

Each seed expands via 4 generators:

  • Temporal prepositions: In the past / Over the last / During the previous
  • Case: lowercase, Title Case
  • Number: singular ↔ plural units
  • Article: bare, with “the”, with “a”

25 seeds → ~2,100 expanded variants

pattern_util -T -o temporal.tsv

Split Temporal / Curated

Phase 3 branching pipeline uses two separate passes:

Temporal uses re.IGNORECASE; curated uses verbatim expansion for controlled matching.
